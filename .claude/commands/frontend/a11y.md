# Accessibility Audit

Audit the Icarus frontend for WCAG 2.1 accessibility compliance.

## Checks

### Semantic HTML
- Are headings in proper order (h1 → h2 → h3, no skipping)?
- Are lists using `<ul>`/`<ol>` instead of divs with bullets?
- Are buttons using `<button>` instead of clickable `<div>`/`<span>`?
- Are links using `<a>` for navigation?
- Is `<main>`, `<nav>`, `<header>`, `<footer>` used appropriately?

### ARIA
- Do interactive elements have proper aria-labels?
- Are `aria-expanded`, `aria-selected`, `aria-current` used for dynamic states?
- Do modals trap focus and have `role="dialog"` + `aria-modal`?
- Are live regions (`aria-live`) used for dynamic content updates?

### Keyboard Navigation
- Can all interactive elements be reached via Tab?
- Do dropdowns/menus support arrow key navigation?
- Is there a visible focus indicator on all focusable elements?
- Can modals be closed with Escape?
- Is there a skip-to-content link?

### Color & Contrast
- Do text colors meet WCAG AA contrast ratio (4.5:1 for normal, 3:1 for large)?
- Is color not the ONLY way to convey information (add icons/text too)?
- Do form error states use more than just red color?

### Forms
- Do all inputs have associated `<label>` elements?
- Are required fields marked with `aria-required`?
- Do error messages reference the field with `aria-describedby`?
- Are fieldsets used to group related inputs?

### Images & Media
- Do all `<img>` tags have meaningful `alt` text?
- Are decorative images using `alt=""` or `aria-hidden`?

### Responsive
- Is content readable at 200% zoom?
- Does the layout work with a minimum viewport of 320px?

## Output
Categorize findings by WCAG level (A, AA, AAA) and impact (CRITICAL/HIGH/MEDIUM/LOW).
Provide code fixes for each issue found.
